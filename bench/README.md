# Cross-engine benchmarks

Micro-benchmarks that pit Soroban's **two calculator engines** against each other on
identical work: the Rust `anzan` crate vs. the Swift `SorobanEngine` library. Both
implement the same Anzan language against one shared [spec](../spec/README.md), so this
is an honest, apples-to-apples comparison of the two implementations — not a language
race with different programs.

The harness is [Gota](https://github.com/alleato-llc/gota), a tiny cross-language
throughput protocol. Gota is **copied, not depended on** (`harness.py`, `report.py`,
`report_template.html`, `rust/src/gota.rs`, `swift/Sources/runner/Gota.swift` are
verbatim copies) — re-copy them when Gota's protocol improves.

## Run it

```sh
python3 bench/run.py                 # build both runners, measure, write results
GOTA_MEASURE=3.0 python3 bench/run.py # longer measure phase for steadier numbers
```

It builds `bench/rust` (`cargo build --release`) and `bench/swift`
(`swift build -c release`), runs each under identical parameters, and writes
[`results.json`](results.json) (raw + provenance) and [`RESULTS.md`](RESULTS.md) (a
table). A missing toolchain is skipped with a note, not an error. For the HTML view:

```sh
python3 bench/report.py bench/results.json -o bench/report.html \
  --title "Soroban engine performance — Rust vs Swift"
```

## What it measures

The op is one **`Calculator.evaluate(line)`** — the fair, symmetric operation both
engines expose. (Swift's `Parser.parse` is `package`-scoped, so full parse+eval via
`evaluate` is the only externally reachable seam anyway — and the honest one to
compare.) The metric is **evaluations per second** (Gota's `op_throughput`, peak of
batches): a single `evaluate()` is far faster than the clock, so the harness batches it,
which auto-calibrates across the ~1000× spread between the fastest and slowest workload
without a shared buffer to fill.

| bench | Anzan workload | hot path |
| --- | --- | --- |
| `arith` | `(123456789.98 / 7.13 + 98765.4) * 2.5 - 1000000 / 3` | BigDecimal division at working precision |
| `fib` | define `fib(n) = if(n <= 2, 1, fib(n-1) + fib(n-2))`; eval `fib(20)` | interpreter dispatch + user-function recursion |
| `reduce` | `sigma_i=1^1000(i^2)` | the indexed ∑ loop + bigint accumulate |
| `transcendental` | `sin(1) + cos(2) + tan(0.5) + atan2(1, 1)` | the f64 `via_double`/`viaDouble` (libm/Double) seam |
| `finance` | `pmt(0.05 / 12, 360, 200000)` | exact `pmt` (power + div) |

Function-defining setup runs untimed; each timed op is a single evaluation, its result
fed to a black hole (`std::hint::black_box` / `withExtendedLifetime`) so the optimizer
can't delete the work. Each workload is also evaluated once, untimed, as a loud guard —
a mistyped line fails there, not silently on the error path.

## Honesty

This is a **same-box, same-run** comparison: both engines are measured back-to-back on
one machine, so the **Rust-vs-Swift ratio** is valid and reproducible anywhere. The
**absolute** evals/sec are hardware- and toolchain-specific — and noisy on shared CI
runners — so treat them as a **trend, not a fixed number**. This follows Gota's
[PROTOCOL](https://github.com/alleato-llc/gota/blob/main/PROTOCOL.md) honesty rule
(*never publish absolute numbers from CI as if they were stable*). The published report
on the site (`/perf.html`, regenerated by `deploy-site.yml`) carries the same caveat.

## Isolation

`bench/rust` is a Cargo project **outside** the `rust/` workspace (its own `target/`,
path-dep on `../../rust/anzan`), and `bench/swift` is a SwiftPM package **outside** the
Engine package (its own `.build/`, path-dep on `../../swift/Engine`). Neither disturbs
the shared workspace/engine builds or their CI. Build artifacts are gitignored.
